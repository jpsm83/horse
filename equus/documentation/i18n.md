# Equus internationalization (i18n)

Equus web UI uses **next-intl** with **English** and **Spanish** (`en`, `es`). The REST API stays locale-agnostic; clients translate stable `error.code` values.

## Locales and routing

| Setting | Value |
|---------|--------|
| Supported locales | `en`, `es` |
| Default | `en` |
| URL prefix | `localePrefix: 'as-needed'` — English at `/`, Spanish at `/es/…` |
| Path segments | English everywhere (`/es/signin`, not translated slugs) |

Configuration lives in `i18n/routing.ts`, `i18n/request.ts`, `proxy.ts`, and `next.config.ts` (next-intl plugin).

## Not-found (404) handling

| Layer | File | Purpose |
|-------|------|---------|
| Proxy | `proxy.ts` | Rewrites unprefixed paths (`/`, `/signin`) into the `[locale]` segment |
| Catch-all | `app/[locale]/[...rest]/page.tsx` | Calls `notFound()` for unknown routes under a locale |
| Localized 404 | `app/[locale]/not-found.tsx` | Translated 404 inside `NextIntlClientProvider` (uses `StatusPageShell`) |
| Root fallback | `app/not-found.tsx` | English-only fallback for non-localized requests (e.g. static files excluded from the proxy matcher); no next-intl client APIs |

## Locale priority (web)

1. **URL `[locale]`** — explicit link or profile save navigation
2. **`NEXT_LOCALE` cookie** — synced from saved `preferredLanguage` via `syncLocaleCookie`
3. **Browser** `Accept-Language` — anonymous first visit or no saved preference
4. **Default** `en`

Once the user has an account, `personalDetails.preferredLanguage` is always set (register, Google session bridge, or profile). The cookie is updated on register, login, Google session sync, and profile save so the choice beats browser detection on return visits.

## User preference

- Stored on `User.personalDetails.preferredLanguage` (`en` | `es`) — **always set** for every user
- Set on **register** from the request `Accept-Language` header when the client does not send `preferredLanguage`
- Set on **Google sign-in** when the account is created (`findOrCreateFromGoogle`) or backfilled on `POST /api/v1/auth/session` from `Accept-Language` if missing
- **Changed only on the profile page** — the preferred-language dropdown; saving updates the user record, syncs `NEXT_LOCALE` via `syncLocaleCookie`, and navigates to the matching locale URL. See [`profile.md`](./profile.md).
- **Login** sets the `NEXT_LOCALE` cookie from saved preference; client redirects if URL locale differs
- There is **no header or global language switcher** — profile is the single place to change app language

## Web development

| Task | Use |
|------|-----|
| UI strings | `useTranslations` / `getTranslations` + `messages/en.json`, `messages/es.json` |
| Links, redirects, router | `@/i18n/navigation` (`Link`, `useRouter`, `usePathname`, `redirect`) |
| Normalize locale (email, API input) | `normalizeLocale` from `i18n/resolveLocale.ts` |
| Sync cookie after preference change | `syncLocaleCookie` (client) or `attachLocaleCookie` (auth routes) |

Do **not** use `next/link` for in-app navigation in the UI layer — use the i18n navigation helpers so locale prefixes stay correct.

## Transactional email

Email templates and link builders use the same two locales. Links use `buildLocalizedAppLink` (`i18n/appLinks.ts`). Senders pass `user.preferredLanguage` when available.

## Folder layout

All i18n code lives under **`i18n/`** (next-intl config + shared helpers). There is no `lib/i18n` or `components/i18n`.

## Mobile (future React Native)

- Share the same JSON message catalogs (`messages/en.json`, `messages/es.json`) — copy or publish from a small shared package
- RN can use `i18next` + `expo-localization` with identical keys
- REST API responses use `error.code` for client-side translation; do not rely on localized server `message` strings in v1

## Tests

- `tests/i18n/resolveLocale.test.ts` — normalization
- `tests/i18n/messages.test.ts` — en/es key parity

## Profile page

The profile form at `app/[locale]/profile/page.tsx` edits all `personalDetails` fields via `PATCH /api/v1/users/me`. Saving `preferredLanguage` updates the user record, syncs the `NEXT_LOCALE` cookie, and updates the active locale URL. This is the **only** UI for changing app language.
