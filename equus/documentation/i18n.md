# Equus internationalization (i18n)

Equus web UI uses **next-intl** with **English** and **Spanish** (`en`, `es`). The REST API stays locale-agnostic; clients translate stable `error.code` values.

## Locales and routing

| Setting | Value |
|---------|--------|
| Supported locales | `en`, `es` |
| Default | `en` |
| URL prefix | `localePrefix: 'as-needed'` — English at `/`, Spanish at `/es/…` |
| Path segments | English everywhere (`/es/signin`, not translated slugs) |

Configuration lives in `i18n/routing.ts`, `i18n/request.ts`, `middleware.ts`, and `next.config.ts` (next-intl plugin).

## Locale priority (web)

1. **URL `[locale]`** — explicit link or language switcher navigation
2. **`NEXT_LOCALE` cookie** — synced from saved `preferredLanguage` via `syncLocaleCookie`
3. **Browser** `Accept-Language` — anonymous first visit or no saved preference
4. **Default** `en`

Once the user chooses a language (register locale, switcher, or profile), `personalDetails.preferredLanguage` is persisted and the cookie is updated so the choice beats browser detection on return visits.

## User preference

- Stored on `User.personalDetails.preferredLanguage` (`en` | `es`)
- Set on **register** from the request `Accept-Language` header (server-side)
- Updated on **language switcher** when authenticated (`PATCH /api/v1/users/me`)
- **Login** sets the `NEXT_LOCALE` cookie from saved preference; client redirects if URL locale differs
- Required for **profile completeness** — register must set it from locale

## Web development

| Task | Use |
|------|-----|
| UI strings | `useTranslations` / `getTranslations` + `messages/en.json`, `messages/es.json` |
| Links, redirects, router | `@/i18n/navigation` (`Link`, `useRouter`, `usePathname`, `redirect`) |
| Normalize locale (email, API input) | `normalizeLocale` from `i18n/resolveLocale.ts` |
| Sync cookie after preference change | `syncLocaleCookie` (client) or `attachLocaleCookie` (auth routes) |

Do **not** use `next/link` for in-app navigation in the UI layer — use the i18n navigation helpers so locale prefixes stay correct.

## Transactional email

Email templates and link builders use the same two locales. Links use `buildLocalizedAppLink` (`i18n/appLinks.ts`). Senders pass `user.preferredLanguage` when available.

## Folder layout

All i18n code lives under **`i18n/`** (next-intl config + shared helpers). UI pieces are plain components (`components/locale-switcher.tsx`). There is no `lib/i18n` or `components/i18n`.

## Mobile (future React Native)

- Share the same JSON message catalogs (`messages/en.json`, `messages/es.json`) — copy or publish from a small shared package
- RN can use `i18next` + `expo-localization` with identical keys
- REST API responses use `error.code` for client-side translation; do not rely on localized server `message` strings in v1
